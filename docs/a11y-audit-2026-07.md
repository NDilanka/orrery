  radial star-point path vs. flat disc + single bar vs. rotating arc pair vs. concentric

**Verdict**: no regression found in what could be checked; the crack read cleanly by

## 3. Keyboard-walk findings

Method: pure-keyboard Tab walks (real `Tab` keypresses, not scripted `.focus()`, to keep
Chromium in genuine keyboard modality) across the system view (Observatory, 24 stops),
the help overlay (5 stops), and TuningConsole (30 stops). For every stop, checked
`document.activeElement.matches(':focus-visible')` plus rendered pixels.

**Every plain `<button>` and text `<input>`** in all three surfaces shows the global
double-ring (`outline: 2px solid var(--border-focus)`, tokens.css:200-203) correctly on
`:focus-visible`. No holes found there.

**Confirmed bug — range slider focus ring never renders**, on both:
- TransportBar's native scrub thumb (`input.scrub.slider`, no `.thumb-brass`)
- All three TuningConsole dials (`input.slider.thumb-brass` × 3 — Ambition, Patience,
  Autonomy)

`:focus-visible` genuinely matches on these elements (verified via
`el.matches(':focus-visible')` = `true` after a real Tab walk), and before/after
screenshots at 3x device-scale, cropped tight on the thumb, show **zero pixel change**
between resting and focused state — no ring, no outline, nothing. This is worse than no
CSS at all: `.slider:focus-visible { outline: none; }` (primitives.css:112-114) actively
suppresses whatever native focus outline Chromium would otherwise draw on a range input,
and the replacement ring never shows up to compensate.

**Root cause, confirmed via `document.styleSheets` introspection** (dumped every parsed
rule in the served primitives.css — 11 rules total): the two rules meant to produce this
ring are **entirely missing from the parsed stylesheet**, even though they're present,
byte for byte, in the source file on disk:

```css
/* primitives.css:82-85 — parses to ZERO rules in Chromium */
.slider::-webkit-slider-thumb,
.slider::-moz-range-thumb {
  transition: box-shadow var(--dur-feedback) var(--ease-standard);
}

/* primitives.css:115-118 — parses to ZERO rules in Chromium */
.slider:focus-visible::-webkit-slider-thumb,
.slider:focus-visible::-moz-range-thumb {
  box-shadow: 0 0 0 2px var(--void), 0 0 0 4px var(--plasma-cyan);
}
```

Chromium/WebView2 doesn't recognize the Firefox-only `::-moz-range-thumb` pseudo-element.
Per the CSS selector spec, an unrecognized/invalid component anywhere in a
comma-separated selector list invalidates the **entire** rule (not just the invalid
branch) — this is the classic "grouping vendor pseudo-elements together silently drops
the whole rule" trap. I isolated this in a minimal standalone HTML page with the exact
same two selectors and confirmed the ring renders perfectly once each vendor variant is
its own separate rule — which is in fact exactly the pattern the file's *other*, working
declaration already uses correctly two rules up (primitives.css:89-98 and 99-106, the
resting `.thumb-brass` thumb style, is split into two standalone rules and works fine).

**Suggested fix** (not applied — primitives.css is out of scope for this pass): split
both broken declarations into standalone `::-webkit-` / `::-moz-` rules, matching the
existing `.thumb-brass` pattern:

```css
.slider::-webkit-slider-thumb { transition: box-shadow var(--dur-feedback) var(--ease-standard); }
.slider::-moz-range-thumb { transition: box-shadow var(--dur-feedback) var(--ease-standard); }

.slider:focus-visible::-webkit-slider-thumb {
  box-shadow: 0 0 0 2px var(--void), 0 0 0 4px var(--plasma-cyan);
}
.slider:focus-visible::-moz-range-thumb {
  box-shadow: 0 0 0 2px var(--void), 0 0 0 4px var(--plasma-cyan);
}
```

Given M4 is about to switch the focus ring to "white double-ring" anyway, whoever picks
this up should fix the selector-grouping bug in the same change rather than patching the
current cyan version and re-patching for M4.

## 4. Prioritized fix list

1. **P0 — range slider focus ring is invisible** (primitives.css:82-85, 115-118). Two
   WCAG 2.4.7-relevant rules are silently dropped by the browser due to grouping
   `::-webkit-` and `::-moz-` pseudo-elements in one comma list. Affects the
   TransportBar scrub and all three TuningConsole dials — every range input in the app.
   Fix: split into standalone per-vendor rules (shown above). Worth doing as part of the
   incoming M4 focus-ring retheme rather than as a standalone patch.
2. **P2 — silhouette coverage gap in this audit, not a product defect**: only the crack
   (failed-dark) rest state was visually re-confirmed in grayscale this pass; the other
   four are sound by code review but weren't independently screenshotted before the dev
   server went down. Re-run once fixtures can force `quota-frost` / `handoff-beacon` /
   `stopped-ember` deliberately, ideally before or right after M4 touches this code path.
3. **P3 — re-run `contrast.test.ts` after M4 lands.** The token *names* the test checks
   are unaffected by M4.1's plan, but nearly every resolved color will change (status
   remap to grayscale, brass/cyan retirement, white focus ring) — today's 26 green
   assertions say nothing about the post-M4 palette.

No other keyboard-walk holes, no other contrast failures.
